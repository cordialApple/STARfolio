# Architecture — Voice (mode A)

Part of the [architecture spec](../architecture.md) · threading: [process-and-ipc.md](process-and-ipc.md)

- **v1 — push-to-talk** ([Stage 6](../stages/stage-06-voice.md)): renderer captures mic (`getUserMedia` + AudioWorklet, 16 kHz mono PCM) while held; on release, PCM goes over IPC to main; whisper.cpp (`smart-whisper`, `base.en`/`small.en`, model downloaded on first use) transcribes the utterance; transcript becomes the user's turn, editable before send. This ships voice without solving streaming — the hold-to-record button *is* the VAD and the endpointer. `getUserMedia`'s `noiseSuppression`/`echoCancellation`/`autoGainControl` constraints cover the "noise suppression" step for close-mic capture; no separate front-end. Guard against whisper's silence-hallucination (empty audio → "Thank you." / "Subtitles by…") by dropping sub-min-duration / below-energy-floor recordings before decode; the editable-before-send transcript is the human backstop.

- **v2 — streaming** ([Stage 6b](../stages/stage-06b-streaming-voice.md), deferrable): whisper is **not** a streaming model — it batch-decodes segments — so "live partial transcript" is a sliding-window re-decode with local-agreement stabilization (chunk + revise), where earlier partials are *revised* as more audio arrives. The real pipeline:
  - **VAD → endpointing**: continuous frames renderer → voice worker; a VAD (Silero ONNX or WebRTC VAD — chosen in the Stage-6b spike, a downloadable model managed like the whisper/embed models if Silero) gates whisper input so silence never reaches the decoder, and endpoints interview-style speech (long thinking pauses must not cut the speaker off mid-answer). Ring buffer in the worker with a backpressure policy for when sustained decode falls behind real-time (RTF must stay < 1.0).
  - **half-duplex turn-taking**: mute capture while OS `speechSynthesis` is speaking, so the open mic doesn't transcribe the interviewer's own question (TTS→mic echo / barge-in). On end-of-utterance the finalized turn goes to the interviewer engine.
  - **typed IPC events** (zod-validated, request-id-scoped like every other stream): `voice:utteranceStart` / `voice:utteranceEnd` from the VAD; `voice:partial { text, stableUpTo }` for revisable partials (`stableUpTo` marks the local-agreement-frozen prefix the UI must not re-flow); a final-transcript event on endpoint. Reuses the existing streaming-via-request-id + abort-by-request-id transport — net-new is the event *shape*, not the mechanism.
  - **data model: zero change** — transcripts persist as session turns (already owned by Stage 5); partials are ephemeral UI state, never stored.

- **Audio is data, never instructions**: streamed/transcribed audio is user turn content fed to the interviewer engine as data, consistent with the grounding posture — a transcript can no more steer the system prompt than an ingested README can.

- **Explicitly rejected**: Web Speech API recognition (routes audio to Google/Azure — breaks the privacy promise) — VAD + local whisper keep all audio on-device. TTS for the interviewer's questions is a toggle on OS `speechSynthesis`; a bundled neural TTS (Piper/Coqui) is out of scope — OS voices suffice for a private practice tool. Speech-to-speech *translation* (NLLB/Marian/multilingual TTS) is out of scope: STARfolio is English-first (`*.en` models), and were non-English practice ever wanted, the path is a multilingual whisper variant + Sonnet (already multilingual) + a locale `speechSynthesis` voice — no dedicated translation model, no new stage.
- **Fallback ladder** if `smart-whisper` fights the Electron ABI: (1) spawn whisper.cpp as a bundled sidecar binary; (2) `nodejs-whisper` CLI wrapper. The Stage-0 spike picks the winner before anything depends on it.
